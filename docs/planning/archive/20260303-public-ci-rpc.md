# RPC: Public CI Baseline for OSS Readiness

## Date
- 2026-03-03

## Status
- Implemented (Baseline + Hardening Pass 3)

## Context
- The repository is moving to OSS posture with tracked secrets/config artifacts removed and `.env.example` added.
- The baseline previously lacked a `.github/workflows` directory; one was added during this pass (`public-ci.yml`).
- For public contributors to trust the repo, CI must validate runtime, quality, and hygiene gates that are deterministic and self-serve.

## Why this matters
- Public CI is the first durable signal of trust for unknown contributors.
- OSS users must be able to verify clean checkouts without local/manual setup tricks.
- CI should prevent configuration regressions, accidental secret leakage, and behavior drift across PRs.

## Design Decision (proposed)
- Use GitHub Actions as the public CI provider.
- Create a single, deterministic baseline workflow set now:
  - `workflow-lint`: validate workflow syntax before quality/test/build execution.
  - `ci`: quality + unit tests.
  - `smoke`: lightweight runtime/bootstrap validation with explicit env examples.
  - `artifact-hygiene`: enforce no tracked runtime artifacts and no banned references.
- Add `workflow_dispatch` support for controlled manual reruns and release-candidate gating.
- Add explicit `timeout-minutes` and pip caching to reduce hanging jobs and cold-start overhead.
- Keep matrix small and stable to avoid flaky infrastructure noise:
  - `python-version: ["3.12", "3.13"]`
  - `quality` runs on `ubuntu-latest` for dependency-safe linting; Windows remains for runtime/bootstrap tests and packaging.

## CI Surface (Initial Scope)
1. Code health
- `python -m ruff check .`
- `python -m black --check src tests`
- `python -m mypy` (optional in phase 1 if dependency profile is stable; otherwise defer)

2. Test gates
- `python -m pytest -q tests/unit` (required)
- `python -m pytest -q tests/integration` (required)
- `python -m pytest -q -m manual tests/manual` (optional, manual lane only)

3. Bootstrap/sanity gate
- `python -m pytest -q tests/unit/runtime/test_bootstrap.py`
- Validate startup behavior on a clean environment with `.env` absent, only `DPOST_*` vars.

4. Artifact hygiene gate
- verify `/.env` and `/build/.env` do not exist in git workspace.
- verify `.gitignore` keeps runtime env files untracked while allowing `.env.example`.
- grep for likely secret-bearing patterns in new/changed files (basic allowlist + extension denylist).

5. Packaging compatibility
- `python -m build` (or equivalent package build task in `pyproject` if available) to catch manifest/import regressions.

## Proposed workflow boundaries
- Use `paths` filter where practical to keep speed:
  - run full matrix on all `src/`, `tests/`, `docs/`, `pyproject.toml`, and workflow files.
- Keep one explicit "required" job per commit:
  - `workflow-lint`
  - `quality`
  - `unit-tests`
  - `integration-tests`
  - `bootstrap-smoke`
  - `hygiene`

## Main Branch Protection Mapping
- Required check contexts for `main`:
  - `workflow-lint`
  - `quality (py3.12)`
  - `quality (py3.13)`
  - `unit-tests (py3.12)`
  - `unit-tests (py3.13)`
  - `integration-tests (py3.12)`
  - `bootstrap-smoke`
  - `artifact-hygiene`
- Canonical payload: `.github/branch-protection/main.required-checks.json`
- Apply script: `scripts/github/set-main-branch-protection.ps1`
- Example:
  - `pwsh -File scripts/github/set-main-branch-protection.ps1 -Repository owner/repo`

## Secret and Config Safety Rules for CI
- Never print raw environment variables in logs.
- `.env` and `build/.env` are never generated by CI; tests that need startup context should use explicit matrix env values.
- Use only non-sensitive placeholder values when seeding defaults in CI.
- Add explicit `DPOST_PC_NAME` and `DPOST_DEVICE_PLUGINS` in workflow env for deterministic runs.

## Definition of Done (for this RPC)
- The CI design document is approved and unambiguous enough for immediate implementation.
- Decision record includes:
  - required jobs and commands,
  - env variable strategy,
  - artifact-hygiene enforcement.
- A follow-up execution checklist references this RPC and is ready to begin implementation.

## Acceptance Criteria (post-implementation)
- Public CI exists and runs on PR and main branch.
- PR merge is blocked unless all required jobs pass.
- `unit-tests`, `integration-tests`, and `hygiene` jobs have zero-tolerance policy for missing env hygiene.
- Runtime bootstrap tests pass in clean env (no `.env` assumptions).
- CI documentation in planning/checklist artifacts links to this RPC.
